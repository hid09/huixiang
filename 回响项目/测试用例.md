## 一、引言

### 1.1 测试目标
本测试方案旨在确保「回响」产品满足需求文档中定义的功能、性能、可用性及模型效果预期，重点验证：
- MVP核心功能（语音/文字记录、日记生成、情绪适配）的正确性和稳定性。
- AI模型（语音转写、情感识别、日记生成）的效果达到产品设计标准。
- 用户体验流畅，界面友好，符合"有温度的对话"理念。
- 系统在国内网络环境下可稳定访问，兼容主流移动浏览器。
- 异常场景下系统具备良好的容错和降级能力。

### 1.2 测试范围
**包含的功能模块**：
- 用户模块（账号登录、注册）
- 语音记录（录音、转写、ASR情感识别、存储）
- 文字记录（输入、情感识别、输入类型判断、存储）
- 日记生成（定时、情绪适配、分层生成策略、v3.0结构）
- 日记列表与详情（含认知变化展示、关键词标签）
- 我的页（统计数据）
- 首页渐进式关怀（连续未记录天数展示）
- 离线同步与本地存储
- 设置页（关于信息）

**不包含的范围**（MVP后迭代）：
- 周/月/季/年度复盘
- AI质量反馈机制（点赞/点踩）
- 行为模式识别

### 1.3 测试类型
- 功能测试
- 模型效果评估（语音转写、情感识别、日记生成）
- 可用性测试
- 性能测试
- 兼容性测试
- 安全测试
- 异常场景测试

### 1.4 测试术语
| 术语 | 说明 |
|------|------|
| MVP | 最小可行产品，指本次上线必须完成的功能集 |
| WER | 词错误率，衡量语音转写准确率 |
| P95 | 95%分位响应时间 |
| FCP | 首次内容绘制 |
| TTI | 可交互时间 |

## 二、测试策略

### 2.1 测试层级
- **单元测试**：前端组件、后端工具函数、AI服务封装（开发自测）。
- **集成测试**：API接口、数据库交互、AI服务调用（手动测试为主）。
- **端到端测试**：核心业务流程（记录→日记生成→查看），手动测试。


### 2.2 测试数据管理
- 使用构造的测试数据集（见附录）进行模型效果评估。
- 功能测试使用模拟数据和真实数据混合。
- 数据库使用SQLite测试实例，测试后清空。

### 2.3 缺陷管理
- 使用GitHub Issues跟踪缺陷，标签：`bug`、`enhancement`、`model-quality`。
- 优先级定义：
  - **P0**：阻塞发布，必须修复。
  - **P1**：严重影响体验，建议修复。
  - **P2**：轻微问题，可后续修复。

## 五、模型效果评估

### 5.1 语音转写准确率评估
**测试集**：20条音频，涵盖不同语速、背景噪音（见附录A）。
**指标**：词错误率（WER）=（替换+删除+插入）/ 参考词总数。
**合格标准**：WER < 15% 为合格。
**测试步骤**：
1. 录制20条音频，保存为期望文本。
2. 运行语音转写服务，获取转写文本。
3. 人工对比转写结果与期望文本，计算准确率。

### 5.2 情感识别效果评估
**测试集**：60条文本记录，包含单一情绪、混合情绪、中性、观点分享误判验证（见附录B）。
**指标**：
- 主导情绪准确率：正确识别主导情绪的比例 > 75%。
- 混合情绪合理性：人工评分（1-5分）平均分 > 3.5。
- **输入类型准确率**：正确识别输入类型的比例 > 80%。
- **观点分享误判率**：观点分享被误判为焦虑/压力的比例 < 10%。
**测试步骤**：
1. 将60条文本输入情感识别API，记录输出（情绪+输入类型）。
2. 对比主导情绪与期望标签，计算准确率。
3. 对比输入类型与期望类型，计算准确率。
4. 重点关注V-01~V-10验证用例，确保观点分享不被误判。
5. 邀请评分员对混合情绪输出合理性打分（1-5分），取平均。

### 5.3 日记生成质量评估
**测试集**：15组每日记录（覆盖0条、1-2条、≥3条场景，见附录C）。
**评估维度**（1-5分）：
- **相关性**：日记内容是否基于用户记录（≥4分及格）。
- **情感共鸣**：语气是否适配当日情绪，有温度（≥4分及格）。
- **结构完整**：v3.0结构完整（5分）。
- **创意性**：小发现、思考碎片是否有洞察（≥3分可接受）。
- **噪音过滤**：有效过滤无效记录（≥4分及格）。
**测试步骤**：
1. 将15组记录输入日记生成服务，生成日记。
2. 邀请3名评分员独立打分。
3. 取各维度平均分，若相关性或情感共鸣 < 4，需优化Prompt。

### 5.4 认知变化检测评估
**测试集**：10组过去与现在记录对（见附录D）。
**指标**：
- 准确率 > 70%
- 召回率 > 60%
**测试步骤**：
1. 输入记录对，获取认知变化检测结果。
2. 对比期望结果，计算准确率和召回率。

## 六、可用性测试

### 6.1 测试目标
- 验证用户能否快速理解"回响"的核心价值。
- 检查录音流程是否顺畅，有无困惑点。
- 确认日记内容和语气是否让用户感到被理解。

### 6.2 测试方法
- **招募用户**：5-8名目标用户（25-35岁，有自我成长意识）。
- **测试任务**：
  1. 首次进入应用，说出你的第一印象。
  2. 录制一段语音（内容自定）。
  3. 查看生成的日记。
  4. 尝试再次记录。
- **观察记录**：操作时长、错误点击、犹豫次数。
- **事后访谈**：收集对录音交互、日记语气、整体感受的评价。

### 6.3 评估指标
- 任务完成率：100%
- 平均任务时间：< 2分钟
- 主观满意度问卷（1-5分）：平均 > 4分

## 九、安全测试

- **数据传输**：确保所有API使用HTTPS，敏感信息（如API Key）不在前端暴露。
- **用户标识**：设备指纹使用哈希处理，不存储原始信息。
- **输入校验**：防止XSS、SQL注入（后端参数校验）。
- **权限控制**：用户只能访问自己的数据，所有接口需做用户ID校验，禁止越权访问。
- **数据隔离**：确保不同用户之间的记录、日记等数据完全隔离。

## 十、异常场景测试

| 场景 | 操作 | 预期 |
|------|------|------|
| 弱网环境 | Chrome模拟3G | 录音上传超时，提示重试；离线存储正常 |
| 服务器500 | 故意触发后端错误 | 前端友好提示，不影响本地记录 |
| AI返回非JSON | 模拟错误格式 | 触发降级，返回模板日记 |
| 并发日记生成 | 同一用户同时请求两次 | 后端保证幂等性，只生成一篇 |
| 录音时切换应用 | 录音中切换到其他应用 | 录音暂停或停止 |
| **越权访问** | 用A用户token请求B用户数据 | 返回403禁止访问，不泄露数据 |
| **并发用户操作** | 多用户同时录音/生成日记 | 各用户数据独立，无交叉污染 |

## 十三、风险评估

| 风险 | 可能性 | 应对措施 |
|------|--------|----------|
| 语音转写准确率低 | 中 | 提前验证，准备备选ASR服务 |
| 日记生成语气生硬 | 中 | 建立Prompt迭代机制，根据反馈优化 |
| 国内访问不稳定 | 中 | 使用国内CDN和云服务，监控各地延迟 |
